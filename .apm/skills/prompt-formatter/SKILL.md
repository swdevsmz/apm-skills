# Prompt Formatter

## 目的

曖昧な依頼、散らかったメモ、長い背景説明を、実行しやすいプロンプトに整理する。

対象は主に次の2系統。

- Codex向け: 実装、調査、編集、レビューなどの実行タスク
- Claude向け: 単発の文章生成、整理、分析、要約、質問応答

## 使う場面

- 「この依頼をAI向けのプロンプトにしてほしい」
- 「雑に書いたメモを整形してほしい」
- 「Codexで強いプロンプトの形にしたい」
- 「Claude用の型とCodex用の型を分けたい」
- 「不足情報を洗い出して、質問も含めて整えてほしい」

## 基本方針

- まず依頼の中心を1文で捉える。目的が曖昧なら、最初にそこを定義する。
- 過剰に長文化しない。モデルは賢い前提で、実行に必要な情報だけを残す。
- `成功条件` と `制約` を特に重視する。ここが弱いと出力がぶれやすい。
- Codex向けでは `タスク` と `成功条件` に加え、`作業モード` を必ず意識する。
- Claude向けでは `役割` `出力形式` `例示` の比重をやや高める。
- 不足情報が致命的なときだけ質問する。質問は 1〜3 個までに絞る。
- 回答必須の項目が分かっているなら、自由文に埋めず見出しで明示する。

## 標準の型

### Codex向け

次の順を基本形にする。

1. `役割`
2. `目的`
3. `タスク`
4. `入力`
5. `制約`
6. `成功条件`
7. `作業モード`
8. `出力形式`
9. `例示`

各項目の考え方:

- `役割`: 必要な場合だけ書く。例: 実装担当、レビュー担当、編集担当
- `目的`: 最終的に何を達成したいか
- `タスク`: 今回AIに何をさせるか。調査だけか、編集までか、実装と検証までかを明示する
- `入力`: 対象ファイル、URL、エラー文、参考資料、前提メモ
- `制約`: 変えてよい範囲、禁止事項、文体、使う技術、避けたい実装
- `成功条件`: 何が満たされたら終わりか。テスト、画面確認、完了状態も含む
- `作業モード`: 不明点があれば質問するのか、合理的に仮定して進めるのか
- `出力形式`: 最後に返す報告の形
- `例示`: 文章トーンや回答フォーマットの参考が必要なときだけ入れる

### Claude向け

次の順を基本形にする。

1. `役割`
2. `目的`
3. `タスク`
4. `成功条件`
5. `制約`
6. `入力`
7. `出力形式`
8. `例示`

Claude向けでは `役割` と `出力形式` と `例示` が効きやすい。一方で、Codexほど強く `作業モード` を要求しないことが多い。

## 差分の考え方

- Claude系は「どう答えるか」を整える色が強い
- Codex系は「どう動いて、どこまで終わらせるか」を整える色が強い
- Codexでは `成功条件` が弱いと途中で止まりやすい
- Codexでは `入力` は文章だけでなく、ファイルやディレクトリの探索対象として扱える
- Claudeでは `例示` が効きやすいが、Codexでは既存コードや既存文書の参照の方が効く場面も多い

## 実行手順

1. 元の依頼を読み、中心目的を1文で要約する。
2. 依頼文から `目的` `タスク` `制約` `入力` `成功条件` を抽出する。
3. 足りない項目を補い、必要なら 1〜3 個だけ質問する。
4. Codex向けか Claude向けかを判断し、型を選ぶ。
5. 冗長な背景説明を削り、実行に必要な粒度へ圧縮する。
6. 最終的なプロンプトを、見出し付きでそのまま使える形に整える。

## 出力ルール

- まず `整形後プロンプト` を出す。
- 次に `補足` として、残っている前提や仮定を短く添える。
- 致命的な不足がある場合だけ、最後に `確認したいこと` を 1〜3 個並べる。
- 不足がなければ質問を作らず、そのまま完成形を返す。

## テンプレート

### Codex向けテンプレート

```markdown
# 役割

# 目的

# タスク

# 入力

# 制約

# 成功条件

# 作業モード

# 出力形式

# 例示
```

### Claude向けテンプレート

```markdown
# 役割

# 目的

# タスク

# 成功条件

# 制約

# 入力

# 出力形式

# 例示
```

## 短い変換例

元の依頼:

```text
READMEを見て、転職エージェントの記事をそれっぽく仕上げて
```

Codex向けの整形例:

```markdown
# 役割

編集担当

# 目的

転職エージェント記事のREADMEを、公開前の下書きとして読める状態まで整える。

# タスク

対象READMEを読んで、構成と本文を整える。必要なら不足箇所を補い、未整理のメモを本文から外す。

# 入力

- README.md

# 制約

- 事実確認できない内容は追加しない
- AIっぽく整いすぎた文体にしない
- 既存の意図を崩さない

# 成功条件

- READMEが自然な記事下書きとして読める
- 未整理メモが残っていない

# 作業モード

不足が軽微なら合理的に補って進める。致命的な不足だけ質問する。

# 出力形式

変更点と、まだ人間確認が必要な箇所だけ短く報告する。
```

## 避けること

- 役割だけ長く書いて、肝心のタスクや成功条件が薄くなること
- 背景説明をそのまま貼り、実行仕様に変換しないこと
- 制約があるのに自由文へ埋もれさせること
- Codex向けなのに `調査だけ` か `実装まで` かを曖昧にすること
- 例示を増やしすぎて、かえって本題が見えなくなること
